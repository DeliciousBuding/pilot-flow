# PilotFlow 答辩 Q&A 指南

> 最后更新：2026-05-01
>
> 本文档覆盖评委可能提出的核心问题，按评分维度组织。每个问题包含标准回答和追问补充。

---

## 维度一：痛点、AI 作用、闭环、Demo 稳定、效率（50%）

### Q1. PilotFlow 解决什么痛点？

**标准回答：**
项目讨论散落在飞书群聊中，关键信息（目标、负责人、截止时间、风险）容易丢失。团队需要在多个飞书功能之间手动切换——先翻聊天记录，再建文档，再建表格，再建任务——每一步都可能遗漏。PilotFlow 让这个过程从"人工搬运"变成"确认后自动执行"。

**追问补充：**
这不是假设痛点。我们自己在准备答辩时就遇到了这个问题：群里讨论了方案，但没有人在第一时间把目标、负责人和交付物沉淀下来。PilotFlow 就是为这个场景设计的——它在讨论发生的同一个群里，把信息结构化并落地。

---

### Q2. AI 在 PilotFlow 里具体做什么？

**标准回答：**
AI 驱动五个环节：理解意图（从群聊文本提取目标、成员、交付物、截止时间、风险）、生成计划（拆解为可执行步骤并分配负责人）、风险检测（识别负责人缺失、截止时间模糊等问题）、工具路由（根据计划自动选择飞书工具并按序执行）、全程记录（记录每个工具调用和产物）。AI 不只是生成文字，它驱动真实飞书对象的创建。

**追问补充：**
我们的 AI 不是通用问答机器人。它有明确的职责边界：提取结构化数据、生成执行计划、选择工具、记录过程。每一步都有 Schema 校验和失败降级，不是"让 AI 自由发挥"。

---

### Q3. 你们说的"闭环"具体是什么？

**标准回答：**
闭环是指从群聊讨论到最终交付的完整链路：群聊意图 -> 生成计划 -> 人工确认 -> 执行飞书工具（创建文档、多维表格、任务、卡片、固定入口）-> 交付总结回传群聊。每一步都有产物，每一步都可追溯。没有确认就不执行，执行失败会记录并降级。

**追问补充：**
闭环的关键不是"能跑通"，而是"每一步都有控制点"。确认机制防止误操作，幂等保护防止重复创建，运行日志让每一步可审计。这是一个有护栏的闭环，不是无限制的自动化。

---

### Q4. Demo 是真实的还是 dry-run？

**标准回答：**
两种模式都支持。dry-run 模式用于本地验证逻辑，不写入飞书；live 模式使用真实飞书 API 创建真实产物。2026-05-01 我们完成了一次 live run（run ID: `run-f7a6ad4e`），创建了真实的飞书文档、多维表格记录、任务、执行计划卡片、风险裁决卡、固定入口消息和交付总结，运行日志保存在 `tmp/runs/latest-live-run.jsonl`。

**追问补充：**
我们对 Demo 状态保持诚实。live run 已验证主路径，但反复团队使用和卡片回调驱动的续跑仍在进行中。我们在 `PRODUCT_REALITY_CHECK.md` 中明确记录了"已验证"和"未完成"的边界，不做超出实际能力的声明。

---

### Q5. PilotFlow 带来什么效率提升？

**标准回答：**
以"启动一个新项目空间"为例，手动操作需要在群聊、文档、表格、任务之间来回切换至少 6 步，容易遗漏。PilotFlow 将这 6 步压缩为一次确认：自动提取要点、生成文档、写入状态表、创建任务、发送入口并固定、生成交付总结。操作者从"逐个创建"变成"审核确认"。

**追问补充：**
效率提升不只是速度，更是可靠性。手动操作容易忘记创建风险记录、忘记固定项目入口、忘记写入截止时间。PilotFlow 的计划模板保证每一步都被覆盖，不会因为人为疏忽而遗漏。

---

### Q6. 如果 AI 提取的信息不准确怎么办？

**标准回答：**
PilotFlow 有两层保护。第一层是 Schema 校验：AI 输出的计划必须通过结构化验证，字段缺失或格式错误会触发降级，不会直接写入飞书。第二层是人工确认：计划以卡片形式发送到群中，用户看到完整计划后才决定是否执行。AI 提取不准不会造成不可逆后果。

**追问补充：**
这也是我们选择"确认门控"架构的原因。AI 的提取能力会随模型升级而提升，但产品层面的安全机制不应该依赖 AI 的准确率。无论 AI 输出什么，写入飞书之前必须经过人工审核。

---

## 维度二：AI 创新、差异化、可复用（25%）

### Q7. OpenClaw 也能做这些，为什么不直接装 OpenClaw？

**标准回答：**
OpenClaw 解决的是"AI 能不能接入飞书、读写对象并执行工具"的问题，是通用 Agent 基础设施。PilotFlow 解决的是"团队在群里讨论出来的东西，怎样被可靠地变成项目运行闭环"的问题，是面向项目运行的产品层。OpenClaw 官方 README 甚至建议把它当私有助手使用，不建议大规模接入企业群聊。PilotFlow 从第一天就是为群协作设计的。

**追问补充：**
底层工具接入会越来越商品化，PilotFlow 的价值不在工具接入，在项目运行产品化。OpenClaw 提供的是 exec approvals（防 shell 风险的底层命令审批），PilotFlow 提供的是业务级确认门控（先预览要创建哪些 Doc/Base/Task，再让 PM 确认）。前者是基础设施，后者是产品。而且 PilotFlow 的架构完全可以把 OpenClaw 视作潜在底座。

---

### Q8. 飞书妙记/飞书项目不是已经能做吗？

**标准回答：**
飞书原生已经把"会议后的纪要与待办"和"项目空间内的流程执行"做得很好。但很多关键决策发生在会前会后的群里，而不是正式会议里；也不是所有重要讨论都会被录成妙记。PilotFlow 补的是中间那段最常见、却最容易失控的协作现场——飞书群里的即时讨论。

**追问补充：**
飞书妙记的入口是会议，飞书项目的入口是工作台。PilotFlow 的入口是群聊。这不是功能重叠，而是入口差异。项目不是从会议开始的，是从群里"先讨论一下"开始的。PilotFlow 抓的是这个起点。

---

### Q9. 和 ChatGPT 直接生成文档有什么区别？

**标准回答：**
ChatGPT 生成的是文本，你需要手动复制到飞书文档、手动建表格、手动建任务。PilotFlow 生成的是飞书真实产物——它直接创建文档、写入多维表格、建立任务、发送卡片、固定入口消息，并且每一步都有记录。从"生成内容"到"创建产物"，这是本质区别。

**追问补充：**
更关键的是 PilotFlow 有确认机制和运行追溯。ChatGPT 生成的内容无法追溯"是谁确认的、基于哪段讨论、写入了哪些对象"。PilotFlow 的 Flight Recorder 记录了完整的执行链路，这对团队协作和项目审计至关重要。

---

### Q10. 你们的确认机制和 OpenClaw 的有什么区别？

**标准回答：**
OpenClaw 的确认机制是 exec approvals，面向 shell/host 操作的安全互锁——命令需要满足 policy、allowlist 和用户批准才会执行，主要防止底层执行风险。PilotFlow 的确认机制是项目语义级审批——先展示将创建的文档、表格、任务、风险卡和固定入口消息的完整预览，再让项目负责人一次性确认。前者防的是"命令执行风险"，防的是"项目操作风险"。

**追问补充：**
这两种确认机制解决不同层次的问题。OpenClaw 的 exec approvals 适合个人助手场景，防止 AI 执行危险命令。PilotFlow 的业务级确认适合团队协作场景，让 PM 在写入发生前看到完整计划，确保项目产物符合预期。两者不冲突，可以叠加。

---

### Q11. 这个项目能推广到其他团队吗？

**标准回答：**
可以。PilotFlow 的架构设计考虑了可扩展性：工具注册表统一管理飞书工具的调用接口，新增工具只需注册即可；编排器按计划有序执行，不依赖特定业务逻辑；Agent 循环有迭代上限，防止失控。任何使用飞书群聊进行项目协作的团队都可以使用，典型场景包括学生团队、产品运营团队、黑客松团队和 AI 原生团队。

**追问补充：**
推广的关键不是代码复用，而是模式复用。"群聊讨论 -> 计划生成 -> 人工确认 -> 自动执行 -> 可追溯"这个闭环模式适用于任何需要从非结构化讨论走向结构化执行的场景。工具可以换，流程可以调，但这个运行模式是通用的。

---

### Q12. 你们的 AI 创新在哪？

**标准回答：**
创新不在单个 AI 能力，而在 AI 驱动的运行架构。我们把 Agent 架构、工具注册表、确认门控和运行追溯四层设计组合在一起，让 AI 不只是"回答问题"，而是"在护栏内执行真实协作工作"。这种"Agent 主驾驶、人类掌舵"的模式，比纯问答或纯自动化都更适合团队协作场景。

**追问补充：**
单独看每一层都不是新发明——Agent 循环、工具调用、审批流、日志记录都有先例。但把它们组合成一个面向飞书群聊场景的项目运行产品，这是 PilotFlow 的设计贡献。创新是组合创新，不是基础研究突破，我们对此保持诚实。

---

## 维度三：AI 技术深度、架构、工程规范（25%）

### Q13. 你们的 AI 技术深度体现在哪？

**标准回答：**
体现在三个层面。第一，结构化提取引擎：从非结构化群聊文本中解析出目标、成员、交付物、截止时间、风险等结构化字段，带 Schema 校验和降级处理。第二，工具注册表：统一管理 9 个飞书工具的调用接口、权限校验和幂等保护，Agent 不直接调用原始 API。第三，有界 Agent 循环：带迭代上限的编排器，防止失控，每次工具调用都有失败事件记录。

**追问补充：**
技术深度不一定要用最前沿的模型。我们的深度在于工程化：如何让 AI 的输出可靠地转化为飞书产物，如何在 AI 出错时优雅降级，如何让每次运行可追溯可审计。这是 Agent 工程问题，不是模型能力问题。

---

### Q14. 架构设计是怎么考虑的？

**标准回答：**
四层架构。第一层是输入层：飞书群聊消息通过 Gateway 进入系统，经过事件去重和 mention 过滤。第二层是规划层：结构化提取引擎解析意图，生成 JSON 格式的项目执行计划。第三层是控制层：确认门控机制，非确认动作只记录不执行。第四层是执行层：工具路由器按计划有序调用飞书工具，写入前预检防重复，失败自动降级。全程通过 JSONL 运行日志记录，支持 Flight Recorder 可视化回放。

**追问补充：**
架构设计的核心原则是"可扩展、可审计、可控制"。工具注册表让新增工具只需注册；运行日志让每一步可追溯；确认门控让副作用需要人工批准。这三层保证了系统不会因为 AI 的不确定性而失控。

---

### Q15. 工程规范体现在哪些方面？

**标准回答：**
五个方面。第一，代码分层：产品 CLI、运行时、Gateway、工具层、文档层严格分离。第二，质量门禁：`npm run pilot:check`、`npm test`、`npm run pilot:audit`、`npm run pilot:doctor` 提供本地质量检查。第三，运行日志：每次运行产生 JSONL 格式的结构化日志，记录工具调用、产物、降级和异常。第四，诚实包装：`PRODUCT_REALITY_CHECK.md` 明确记录已验证和未完成的边界，不做超出实际能力的声明。第五，幂等保护：重复运行检测防止意外创建重复产物。

**追问补充：**
工程规范的核心是"可维护、可验证、可诚实"。代码分层让后续开发者能快速理解边界；质量门禁让每次提交都有基本保障；运行日志让问题可追溯；诚实包装让团队和评委知道真实状态，不被过度乐观的文档误导。

---

### Q16. Flight Recorder 是什么？

**标准回答：**
Flight Recorder 是 PilotFlow 的运行追溯系统。每次运行都会产生 JSONL 格式的日志，记录每个工具调用的输入、输出、状态、产物链接和异常信息。这个日志可以渲染为本地 HTML 页面，可视化展示整个执行过程。它的定位是业务级审计——记录"谁发起、确认了什么、写入了哪些对象"，而不是工程级的 debug trace。

**追问补充：**
Flight Recorder 的价值在于可追责、可理解、可恢复。当项目产物出现问题时，可以通过 Flight Recorder 回溯：是哪个步骤出了问题、AI 提取了什么、用户确认了什么、哪个工具调用失败了。这对团队协作和项目审计至关重要。

---

### Q17. 你们怎么保证 AI 输出的可靠性？

**标准回答：**
三层保障。第一层是 Schema 校验：AI 输出的项目计划必须通过 JSON Schema 验证，字段缺失或格式错误直接拒绝。第二层是计划验证回退：当计划不安全或不完整时，系统进入澄清流程而不是直接执行。第三层是确认门控：即使计划通过校验，也需要人工确认后才执行写入操作。AI 的准确率是改进目标，但产品安全不依赖于它。

**追问补充：**
我们没有假装 AI 总是对的。工具失败会被记录和展示，Agent 不会假装失败的写入成功了。这种"不掩盖失败"的设计理念，比追求 AI 的高准确率更务实。

---

### Q18. 这个项目的商业化前景？

**标准回答：**
短期看，PilotFlow 适合作为飞书生态内的轻量项目管理补充，面向学生团队、黑客松团队和中小产品团队。中期看，如果飞书开放平台的 Bot 能力成熟，可以做成飞书应用市场上架。长期看，"群聊讨论 -> 计划 -> 确认 -> 执行 -> 追溯"这个运行模式可以扩展到其他 IM 平台（钉钉、企业微信），但当前阶段聚焦飞书。

**追问补充：**
商业化的核心挑战不是技术，而是飞书平台的 Bot 生态成熟度。当前飞书的卡片回调、群公告 API 还在演进中，这限制了全自动化的可行性。但我们相信底层工具接入会越来越商品化，项目运行产品化的价值会越来越明显。

---

### Q19. 和直接用飞书 OpenAPI 有什么区别？

**标准回答：**
飞书 OpenAPI 是原始接口，需要开发者自己处理认证、错误处理、幂等、重试等问题。PilotFlow 在 OpenAPI 之上构建了工具注册表，统一管理调用接口、权限校验和幂等保护；更关键的是，PilotFlow 在工具调用之上构建了 AI 编排层——AI 理解意图、生成计划、选择工具、按序执行，开发者不需要手动编排每个 API 调用。

**追问补充：**
飞书 CLI 被官方定义为"AI Agent 的标准化执行入口"，专门为 Agent 设计，能在出错时告诉 AI 怎么修复。PilotFlow 完全可以将 CLI 作为潜在底座，我们的价值在 CLI 之上的项目运行逻辑。

---

### Q20. 你们最大的技术风险是什么？

**标准回答：**
两个风险。第一，飞书平台能力的边界：当前测试群的群公告 API 返回 docx 类型阻断，卡片回调的完整端到端验证还在进行中。第二，AI 规划的可靠性：当前 LLM 规划层是脚手架级别，真实规划成功率还不是产品承诺。我们通过确认门控和 Schema 校验来对冲这两个风险——即使平台或 AI 出问题，也不会造成不可逆的副作用。

**追问补充：**
我们选择"诚实包装"策略来管理风险。在 `PRODUCT_REALITY_CHECK.md` 中明确记录了"已验证"和"未完成"的边界。评委和用户看到的是真实状态，而不是过度乐观的宣传。这种透明度本身就是一种工程规范。
