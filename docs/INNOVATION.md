# PilotFlow 创新点

PilotFlow 不打算再做一个飞书 API 封装。它要解决的问题更窄：群聊里经常出现目标、承诺、风险、截止时间，但还没人把它整理成项目。PilotFlow 把这一段从模糊讨论到明确项目的过程做成产品。

## 产品创新

### 1. 从群聊直接启动项目

项目最早出现的位置是群聊里的一句话——"这个谁来做"、"下周要交"、"这里可能卡住"。PilotFlow 接住这一句，把它解读成包含目标、成员、交付物和截止时间的执行计划，再发卡片到群里等用户确认，最后落地成飞书产物。

PilotFlow 不是替代项目管理工具的，是在项目还没成型之前补上"识别和启动"这一段。

### 2. AI 的判断要显式传

项目模板、看板视图、筛选条件、风险等级、分页这些判断都由 Hermes Agent 用结构化字段交给工具，PilotFlow 不靠关键词替用户拍板。

字段不全的时候，工具不去猜，直接让 Agent 回到群里追问。这一步把"AI 想了什么"和"产品做了什么"分开，让用户能看清楚每一步。

### 3. 写入飞书前必须用户确认

创建文档、Base 记录、任务、日历事件、提醒和权限都会改变用户的工作空间。PilotFlow 不会在读完一句话之后立即执行。

它先把计划发成卡片，用户确认或取消之后再决定是否执行。同样的规则适用于移除成员、归档项目这类高风险动作。

### 4. 群聊里跑完整个项目

项目建好之后，用户继续在同一个群里查状态、改截止时间、记录风险、催办、标记完成、归档。看板卡、详情卡、简报卡都可以直接点按钮操作。

不需要切到飞书项目或别的工具，群聊本身就是项目入口。

## 工程创新

### 5. 卡片按钮走服务端短期引用

互动卡片的按钮只携带一个短期、随机、不可解读的引用。它不带项目名、不带群 ID、不带操作内容。

PilotFlow 在服务端用这个引用查回真实上下文，操作成功后把引用消费掉，操作失败时保留引用让用户重试。对用户来说，按钮失败之后再点一次就能继续；对系统来说，伪造的按钮 payload 走不通。

### 6. 重启之后接得回去

PilotFlow 把项目状态分两层存：公开摘要不含群号、成员 ID、文档链接、token；私有引用本地保存且不进 git。

状态写入有文件锁和版本字段，所以 Hermes 重启之后，看板、详情、催办、待办、待续按钮都能恢复，用户感觉不到中断。

## 后续工作

- 读取历史项目模式，让 Agent 在新项目里建议常用的成员、交付物和工期。
- 把执行层逐步拆成更小的模块，但保持对外的工具接口稳定。
- 在飞书项目（Meego）OpenAPI 接入之后，把它作为立项后的权威后端，PilotFlow 仍然负责群聊侧的启动和巡检。
